Reasoning GP · one-time setup

Connect the AI — so Ask and Scribe work on your live site

Follow this once. No coding, no command line — everything is done on the Cloudflare website. About 20 minutes.

First, why it isn't working

Your website (the pages at gpreasoning.uk) is just files. It cannot run AI by itself — a website can't safely hold your secret AI key.

The AI needs a small separate program — a Cloudflare Worker — that lives on Cloudflare's servers, holds your key privately, and answers the AI requests. The code for it is already written for you (it's in your project's backend/worker.js). You just have to publish it once and tell your site where it lives. That's the whole job.

Scribe's microphone is different. The mic is a built-in browser feature and needs no backend — that's why it seemed to "work". But turning your speech into a written note, and everything in Ask, needs the Worker below.

What you need open in tabs

  • Anthropic Console — console.anthropic.com (for the AI key)
  • Cloudflare dashboard — dash.cloudflare.com (free account)
  • Your project's backend/worker.js file open, ready to copy its contents

Part 1 — Get your AI key

  1. Sign in at console.anthropic.com. Go to Billing and add a little credit (e.g. £10) and set a monthly spend limit so it can never surprise you.
  2. Go to API keys → Create key. Copy the key (it starts sk-ant-…) and paste it somewhere safe for a minute. You only see it once.

Part 2 — Publish the Worker on Cloudflare

  1. At dash.cloudflare.com, open Workers & Pages → Create → Create Worker. Give it a name like reasoning-gp-api, then click Deploy (it deploys a placeholder — that's fine).
  2. Click Edit code. Select all the placeholder code and delete it. Open your backend/worker.js, copy everything, paste it in, then click Deploy (top right).
  3. Create the two databases. In a new tab go to Workers & Pages → KV → Create a namespace. Make one called USERS. Do it again for TOKENS. (These store accounts & logins.)
  4. Connect the databases to the Worker. Open your Worker → Settings → Bindings → Add → KV namespace. Add two:
    • Variable name USERS → select the USERS namespace
    • Variable name TOKENS → select the TOKENS namespace
  5. Add your secret AI key. Same Settings → Variables and Secrets → Add. Type Secret:
    AI_API_KEY = your sk-ant-… key
  6. Add two plain settings. Same place, type Text (not secret):
    ALLOWED_ORIGINhttps://gpreasoning.uk  — your site address, no slash at the end
    AI_MODELclaude-sonnet-4-6  — best quality/price. For more volume per £, use claude-haiku-4-5
    Click Deploy again so the new settings take effect.
  7. Copy your Worker's address. On the Worker's overview page it shows a URL like https://reasoning-gp-api.yourname.workers.dev. Copy it.
  8. Test it. Paste that URL in a browser and add /api/health on the end. You should see {"ok":true}. That means the Worker is alive.
Almost there. The engine is now running. One last step tells your website to use it.

Part 3 — Point your site at the Worker

  1. Open assets/site.js in your project. Near the very top find:
    workerUrl: '',
    Paste your Worker address inside the quotes (no slash at the end):
    workerUrl: 'https://reasoning-gp-api.yourname.workers.dev',
  2. Publish the change the same way you publish your site (push to GitHub). Then hard-refresh the Ask page (Ctrl+Shift+R) so the browser loads the new file.
  3. Open Ask and ask a question. It should now answer. 🎉

Part 4 — (Optional) Turn on semantic search recommended

By default Ask finds the right library topic by matching words (with a big clinical-synonym map, so "SOB in a smoker" still finds breathlessness/COPD). Semantic search makes it understand meaning — so even loose, colloquial questions land on the right note. It's free (Cloudflare Workers AI) and Ask works fine without it, so this is safe to add any time.

  1. Open your Worker → Settings → Bindings → Add → Workers AI. Set the variable name to exactly AI and save.
  2. Click Deploy. That's it — Ask detects the binding and switches on semantic search automatically. If you ever remove it, Ask quietly goes back to word-matching. Nothing breaks either way.
Note: semantic search only runs on your live site (where the Worker lives), not inside the Claude builder preview. The first time someone uses Ask after enabling it, their browser quietly indexes the library once, then it's instant.

Important: who is allowed to use the AI?

The Worker is currently set so that only signed-in, paying subscribers can use the AI. That means straight after setup, Ask will still say it can't answer unless the person is logged in on a paid plan — and you haven't set up payments yet. Choose one:

Option A — Free for everyone (fastest to see it working)

In backend/worker.js, find the function aiProxy (near the bottom) and delete these three lines, then re-paste & Deploy the Worker:

let user; try { user = await userFromToken(request, env); } catch (e) { return json({ error: 'sign_in_required' }, 401, cors); } if (!user.tier || user.tier === 'bronze') return json({ error: 'subscription_required' }, 402, cors);

Removing them lets anyone use the AI. Simple, but your AI credit pays for all usage — keep your Anthropic spend limit on.

Option B — Free account required (no payment)

Keep the sign-in lines, delete only the last line (the bronze / subscription_required one). Anyone who registers a free account gets the AI.

Option C — Keep it paid

Leave it as-is and complete the PayPal steps in backend/README.md (Steps 6–8). Subscribers get unlocked automatically.

Part 4 — (Optional) turn on semantic search smarter Ask

This makes Ask understand meaning, not just keywords — so "can't catch my breath in a smoker" finds the breathlessness/COPD notes even without matching words. It uses Cloudflare's free built-in AI, so it costs nothing extra. If you skip this, Ask still works well using keyword + synonym matching.

  1. Open your Worker → Settings → Bindings → Add → Workers AI. Set the variable name to exactly AI and save.
  2. Click Deploy. That's it — the site detects it automatically. (The first few questions per device quietly build a semantic index, cached in the browser; if the binding is ever missing, Ask silently falls back to keyword search — it never breaks.)
Already in your config. The [ai] binding = "AI" line is already in wrangler.toml, so if you deploy by CLI it's automatic. This step is only needed if you set the Worker up through the Cloudflare website.

Part 5 — Live web search already on

Ask can also check current guidance live on the web — restricted to trusted UK sources only (NICE / CKS / BNF, SIGN, BMJ Best Practice, GOV.UK / MHRA / UKHSA, and UK specialist societies). It searches only when the answer depends on something recent it isn't sure about, and shows the pages it used under the answer as "Live sources". Nothing to set up — it's on by default.

  • Cost: about 1p per search on top of the answer — pennies a month at typical usage, already covered by the £20 estimate.
  • To turn it off: Worker → Settings → Variables → add a Text variable WEB_SEARCH = off, then Deploy.

Part 6 — The quality flywheel owner tools

Three things keep Ask provably accurate and cheap. They're all built in — you just need to add one variable:

  1. Set the admin email. Worker → Settings → Variables → add a Text variable ADMIN_EMAIL = the email of your account on the site, then Deploy. This unlocks the owner console for you (and nobody else).
  2. Open the Quality Console at tools/ask-quality.html (deliberately not linked in the public menus — bookmark it). Three tabs:
    • 🧪 Test bank — ~45 gold-standard questions (doses, thresholds, 2WW criteria) run through the exact pipeline users get, each answer checked for its must-have facts. Run it after any change to the library or settings.
    • 💬 User feedback — every 👍/👎 users leave on an answer. Review weekly; each 👎 shows you which library note to fix.
    • ⚡ Cached answers — popular questions are answered once and then served instantly (zero AI cost). Approve the good ones to make them permanent, hand-vetted answers; delete weak ones to regenerate.

Part 7 — Members-only access & payments owner setup

The site can be locked so only paying members (and people you comp) get in. All of it runs on the same Worker — no Firebase needed; your accounts already sync across computers.

  1. Turn the paywall on. In assets/site.js, RGP_CONFIG.paywall is true and paywallHosts lists your live domains. Previews and local copies are never locked — only the real site gates. Set paywall:false to reopen everything.
  2. Tiers. Admin → everything · Platinum → everything · Silver → clinic tools · Gold → SCA prep · signed-out / bronze → locked.
  3. Take payments. Card & Apple Pay go through PayPal (a card can't be charged straight into a bank account). Set RGP_CONFIG.paypalClientId + paypalPlans in site.js and the Subscribe buttons appear on the pricing cards; the Worker verifies the webhook and grants the tier automatically.
  4. Get the money into Monzo Business. The processor collects the payment; you then route payouts to Monzo. In PayPal, add your Monzo account number + sort code as the linked withdrawal bank (or, if you use Stripe instead, set Monzo as the payout bank). Funds land in Monzo automatically.
  5. Free entry / discount codes. Set ADMIN_EMAIL on the Worker (Part 6) to your login email, then open tools/access-codes.html (not linked in the public menu — bookmark it). Create codes that grant a tier for a set number of days, with a max number of uses and optional expiry. Share a code; the person redeems it under Account → Redeem a code for full comped access — no payment taken.

Keeping it under £20/month

Hosting (Cloudflare) is free. The only cost is the AI itself, paid per question to Anthropic. With the recommended Claude Sonnet 4.6 model, a typical question costs about 2p — so £20 covers roughly 900 questions a month. Switching the AI_MODEL to claude-haiku-4-5 costs ~0.7p each (~2,700/month) at slightly lower quality.

Hard ceiling: in the Anthropic Console → Billing, load prepaid credit (e.g. $25) and turn auto-reload OFF. The AI simply stops when the credit runs out — it can never bill you more than you've added.

If something's not right

  • Ask still says "isn't connected": the workerUrl in site.js is empty, mistyped, or the new file hasn't published yet. Re-check Part 3 and hard-refresh.
  • Answers fail with an error, not the "isn't connected" message: the Worker is reached but rejecting — almost always the access gate above (Option A/B), or ALLOWED_ORIGIN doesn't exactly match https://gpreasoning.uk.
  • /api/health doesn't show {"ok":true}: the Worker code wasn't pasted/deployed. Redo Part 2, steps 1–2.
